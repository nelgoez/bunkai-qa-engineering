## 1. Executive Risk Map

Bunkai's attack surface is a tight loop: **auth gates everything**, **ATCs are the product value**, and the **Run state machine is the daily workflow**. Break any of these three and the system is either wide open, valueless, or unusable. Two qualities make testing harder here than in a typical SaaS: every table is RLS-gated (per-tenant isolation lives in the database, not the app layer), and the same endpoints serve humans, AI agents, and CI runners ‚Äî a shape bug in one mode is a shape bug in all three.

| Priority  | Flow                                                | Why it matters                                                                 | Depends on / Affects                            |
| --------- | --------------------------------------------------- | ------------------------------------------------------------------------------ | ----------------------------------------------- |
| CRITICAL  | Auth, session & RBAC (JWT + Bearer PAT + RLS)       | Security perimeter. Two auth paths double the attack surface. RLS is the **only** data-isolation layer between tenants ‚Äî no fallback. | Everything gated by login; multi-tenancy; every endpoint |
| CRITICAL  | ATC creation + mandatory provenance (US ‚Üí AC ‚Üí ATC) | Bunkai's differentiator. Transaction spans 4 tables + M:N join. Validation rejects orphan ATCs at the constraint layer ‚Äî if it fails silently, the data model's promise is broken. | Setup flow, Test chain, edit propagation, every Run |
| CRITICAL  | ATC edit propagation (one-edit-many-tests)           | Version increments on edit. Tests reference ATCs, they don't copy them ‚Äî a corruption here silently poisons every downstream Test that chains the ATC. | Test chain integrity, manual/agentic Runs |
| HIGH      | Run lifecycle (start ‚Üí step-report ‚Üí abort/finish)   | Core daily workflow for humans and agents. State machine must be **monotonic** (terminal = no more writes) and recomputes parent statuses in-transaction. | Bugs, Realtime, dashboards, Jira sync |
| HIGH      | Setup flow (Workspace ‚Üí Project ‚Üí Module ‚Üí US ‚Üí AC) | Adoption funnel ‚Äî the first 10 minutes. If sign-up or workspace creation fails, no other journey runs. | Everything downstream |

## 2. What to Test First and Why

### 2.1 Auth, Session & RBAC

**Why it matters.** Auth is the security perimeter. If RLS fails, tenant A sees tenant B's data. If Bearer PAT verification has a timing hole, an expired token keeps working. If role inheritance breaks, a `viewer` can write. There's no recovery from an RLS breach ‚Äî data leaked, trust gone. Two consumer classes (browser JWT + Bearer PAT) mean every endpoint needs testing under both auth modes.

**What commonly breaks.**
- RLS policies that pass with `NULL workspace_id` ‚Äî every table must reject rows where `workspace_id IS NULL` for non-service-role queries.
- Token scopes not AND-ed with role ‚Äî a `member` who mints a read-only PAT should not be able to create ATCs even though the underlying user can.
- `revoked_at` check missing in the Bearer middleware ‚Äî the SHA-256 lookup returns the row, but no column filter excludes revoked tokens.
- Idempotency-Key bypassing auth ‚Äî a caller without a valid token replaying a known key should still hit 401, not the cached response.
- `workspace_id` not attached to request context when JWT is valid but user has no active workspace ‚Äî null workspace should surface a clean 403, not a 500.

**Dependencies.** Everything. No auth = no flow works. Specifically: `middleware.ts` ‚Üí every route handler ‚Üí Supabase RLS policies ‚Üí `workspace_members` table.

**What an experienced QA would check.**
- Run every endpoint under all four roles (`viewer`, `member`, `admin`, `owner`) with both JWT and Bearer PAT, verifying write endpoints reject `viewer` and `member` with read-only token.
- Create a PAT, revoke it via `DELETE /auth/tokens/{id}`, fire the next request within the same second ‚Äî response must be 401 `TOKEN_REVOKED`, not cached 200.
- Cross-tenant isolation: create data in workspace A, switch to workspace B, attempt direct `GET` on workspace A's entity IDs ‚Äî every endpoint must return 403 or 404 (never 200 with data).
- Idempotency-Key replay across tenants: create a `POST /runs` with key `X` in workspace A, replay the same key in workspace B ‚Äî must return 403 if workspace-scoped, not the workspace A `run_id`.

### 2.2 ATC Creation + Mandatory Provenance

**Why it matters.** This is Bunkai's structural rule: every ATC carries `user_story_id` AND ‚â•1 row in `atc_acceptance_criteria`. If this constraint fails, orphan ATCs exist ‚Äî the product's pedagogy (anchored traceability) is a lie. The create transaction spans `atcs` + `atc_steps` + `atc_assertions` + `atc_acceptance_criteria` + `activity_log`. A partial write in any of these leaves the system in an inconsistent state.

**What commonly breaks.**
- `atc_acceptance_criteria` join silently skipped ‚Äî ATC inserts but the M:N link is missing. The read endpoint returns `acceptance_criteria: []`, the UI shows "orphan ATC," and the user doesn't know why.
- AC belonging to a different User Story than the ATC's `user_story_id` ‚Äî the validation must reject cross-US ACs (data-map ¬ß3: "acceptance criterion belonging to that US").
- Step position uniqueness ‚Äî two steps at position `1`, or a gap in positions (1, 3, 5), breaks the runner's sequential-advance logic.
- Layer enum not validated ‚Äî `layer: "browser"` passes instead of `UI|API|Unit` ‚Äî downstream filters break silently.
- Tag limit > 10 bypassed ‚Äî array length check missing in Zod schema.

**Dependencies.** User Story and AC must exist first (Flow 1, Steps 5‚Äì6). ATC search (`tsvector` GIN index) depends on correct insert trigger. ATC edit propagation depends on `version` counter increment.

**What an experienced QA would check.**
- Create an ATC with `acceptance_criterion_ids` pointing to an AC that belongs to a **different** User Story ‚Üí must return 400 with a code like `AC_MISMATCH_US`, not 201 with a partial write.
- Fire two concurrent `POST /atcs` for the same module with overlapping step positions ‚Äî verify position uniqueness survives race conditions.
- Create ATC with `layer` = `"invalid"`, `null`, missing, and "" ‚Üí all must reject; only `UI`, `API`, `Unit` accepted.
- Query the ATC immediately via `GET /atcs/{id}?expand=acceptance_criteria` ‚Üí verify the M:N join returned, all ACs present, no extras.

### 2.3 ATC Edit Propagation (One-Edit-Many-Tests)

**Why it matters.** Tests store `test_steps.atc_id` (a foreign key, not a copy). When an ATC is edited, every Test that chains it sees the new version automatically. This is the entire "one-edit-many-tests" promise. A version-counter skip, a cascade-replace bug that drops steps, or a stale reference left in a Test's chain ‚Äî any of these corrupts test execution silently, and the corruption is invisible until someone runs the Test.

**What commonly breaks.**
- `version` counter not incremented on edit ‚Äî consumers (CLI, agents, UI) cache by version; stale version means stale content at runtime.
- Cascade replace of `atc_steps` deletes all rows before re-inserting, but a mid-transaction error leaves zero steps ‚Üí ATC becomes empty.
- Test chain still references the ATC but `affected_test_ids` not emitted in the `atc.updated` event ‚Üí Realtime subscribers don't refetch the Test expanded view.
- Concurrent edit by two users on the same ATC ‚Äî last-write-wins without optimistic-lock check ‚Üí one user's edit silently clobbered.

**Dependencies.** Runs that snapshot ATC content at `run_atcs` creation time are **not** affected (they store a point-in-time copy). But the Test contract that agents fetch via `?expand=atcs.steps` **is** affected ‚Äî an agent fetching mid-edit could get a partially updated chain.

**What an experienced QA would check.**
- Create Test with ATC {steps=[A,B]}, edit ATC to {steps=[X,Y,Z]}, fetch Test via `?expand=atcs.steps` ‚Üí assert steps are X,Y,Z (not A,B).
- Edit an ATC used by 3 different Tests, then fetch all 3 Tests expanded ‚Üí all 3 reflect the edit.
- Start a Run mid-ATC-edit ‚Äî the Run skeleton at `run_atcs` snapshots the pre-edit state; changing the ATC after Run creation should NOT mutate the Run.
- Realtime broadcast: open two browser tabs, edit ATC in tab A ‚Üí tab B's tree view reflects the change within 3s without manual refresh.

### 2.4 Run Lifecycle

**Why it matters.** The Run is the daily workflow for every persona. The state machine has a hard rule: **terminal states are absorbing** ‚Äî once `passed|failed|aborted`, no step writes are accepted. Step-result recomputation (`run_atcs.status` derived from children, never client-written) must happen in the same transaction as the leaf update. A desync between `run_steps` and `run_atcs.status` makes the runner UI lie to the user.

**What commonly breaks.**
- Step result posted to a terminal Run ‚Üí must return 409 or 400 (data-map ¬ß3: "no further step updates are accepted"). If this gate is missing, a finished Run's status can flip back.
- `run_atcs.status` not recomputed correctly on mixed results ‚Üí 3 pass + 1 block = `block` (not `pass`, not `running` ‚Äî child-status priority: fail > block > pass > pending).
- `runs.progress_pct` division by zero when a Test has zero steps or steps are all skipped ‚Äî must return 0 or 100, not NaN/null.
- Idempotency key collision window: start Run with key `daily-001`, abort it, retry start with same key within 24h ‚Üí should return the **cached** response (the original `run_id`), not create a second Run.
- Sweeper cron marks a `running` Run as `aborted` after 4h, but a step-result write arrives between the sweeper's SELECT and UPDATE ‚Üí race condition: Run becomes terminal with a step still reporting.

**Dependencies.** Realtime broadcasts on `runs`, `run_atcs`, `run_steps`. Defect heatmap (if a Bug is filed). Jira sync worker (if integration is on).

**What an experienced QA would check.**
- Run through the full lifecycle: start ‚Üí mark 3 steps pass ‚Üí mark step 4 `fail` ‚Üí verify `run_atcs.status=fail` before marking remaining steps ‚Üí file Bug ‚Üí abort ‚Üí verify remaining steps swept to `skipped` in the same response.
- Post a step result to a Run that was aborted 1 second ago ‚Üí must get 409, not 200, not 500.
- Start two Runs on the same Test with different `Idempotency-Key` values simultaneously ‚Üí both succeed, separate `run_id`s returned.
- Start a Run, mark all steps pass, finish ‚Üí re-fetch the Run ‚Üí status must be `passed`, `finished_at` set, `progress_pct=100`.

### 2.5 Setup Flow (Adoption Funnel)

**Why it matters.** New user onboarding is the funnel. OAuth callback ‚Üí first workspace ‚Üí first project ‚Üí first module tree ‚Üí first US/AC ‚Üí first ATC. Each step is a gate; break any one and the user bounces. The path has multiple external dependencies (Supabase Auth, optional Jira import).

**What commonly breaks.**
- First verified login doesn't auto-create a default personal workspace ({{PROJECT_KEY}}-001) ‚Üí user lands on an empty state with no CTA.
- Workspace slug collision on reserved words (SRS lists a reserved-list guard) ‚Äî if missing, someone claims `admin` or `api` as a slug.
- Module circular-parent check only works at depth 1 ‚Äî moving a module under its own descendant at depth 4 creates a loop that the recursive CTE in the tree view can't resolve.
- Jira import dedup by `external_id` fails on re-run ‚Üí duplicate US rows.

**Dependencies.** Auth (Step 1). Module tree (Step 3). US/AC (Steps 4‚Äì5). ATC (Step 6).

**What an experienced QA would check.**
- Complete the full setup journey end-to-end without hitting a single 500 or blank screen ‚Äî time it (target: <10 minutes per PRD).
- Sign in with GitHub OAuth, revoke GitHub access from the GitHub side, attempt to use the existing JWT ‚Üí token should expire, user prompted to re-auth.
- Create a module tree at depth exactly 6 ‚Üí succeeds. Attempt depth 7 ‚Üí 400 with `MODULE_DEPTH_EXCEEDED`.

### 2.6 Realtime Fan-Out

**Why it matters.** Without Realtime, the runner UI polls every 3‚Äì5 seconds. With Realtime, Mateo's dashboard shows live updates as Elena marks steps. The channel is keyed per `project_id` ‚Äî a mis-keyed subscription means data leaks across projects or updates never arrive.

**What commonly breaks.**
- Channel subscription attaches to wrong `project_id` ‚Üí tab A's runner UI receives updates from tab B's different project.
- Realtime broadcast fires on INSERT but not on UPDATE ‚Üí step-result changes (`pending‚Üípass`) never reach the UI.
- WebSocket reconnect drops events during the gap ‚Üí runner UI shows stale status until manual refresh.
- High-frequency step-result posting (agent mode, 20 steps/second) overwhelms the Realtime channel ‚Üí message coalescing drops intermediate states but the final state must always arrive.

**Dependencies.** Postgres logical replication (Supabase-managed). Runner UI. Dashboard.

**What an experienced QA would check.**
- Open two browser tabs on the same Run. In tab A, mark step 1 `pass`. Tab B's runner must update within 3s without any manual action.
- File a Bug from the runner in tab A ‚Üí tab B's defect heatmap reflects the new count within 5s.
- Kill the WebSocket connection (simulate network drop), mark 3 steps, reconnect ‚Üí all 3 final states arrive, not just the last one.

## 3. State Machines That Matter

### 3.1 `runs.status` ‚Äî Run Lifecycle

**Why the transitions matter.** A Run that can be reopened after terminal status breaks every downstream assumption: dashboards show stale "running" counts, agents retry steps on a finished Run, the heatmap double-counts. Terminal states must be **absorbing and irreversible**.

**Transitions most likely to be broken.**
- `running ‚Üí aborted` receiving a step-result concurrently with the abort ‚Üí the sweep to `skipped` races against the explicit step update.
- `POST /finish` with a valid `status=failed` but one `run_atc` is still `pending` ‚Üí should the `/finish` sweep pending to skipped and mark the Run failed, or reject?

**Terminal / forbidden states to guard.**
- A terminal Run (`passed|failed|aborted`) receiving `POST .../result` must return 409 with `RUN_TERMINAL`.
- A `running` Run with zero reported steps should not transition to `passed` ‚Äî it should stay `running` until explicit finish.
- Sweeper cron must never abort a Run that finished normally between its SELECT and UPDATE.

**How corruption would be detected.** `runs.progress_pct` > 100, `finished_at` set while `status=running`, or a Run with `status=passed` but a `run_atc` child with `status=fail`. Audit query: `SELECT * FROM runs WHERE status IN ('passed','failed','aborted') AND finished_at IS NULL`.

---

### 3.2 `bugs.status` ‚Äî Bug Lifecycle

**Why the transitions matter.** The Bug is the native defect artifact. Its lifecycle feeds the heatmap and, if Jira is connected, the sync worker. Reopening (`closed ‚Üí open`) is explicitly allowed and must be tracked as a regression metric.

**Transitions most likely to be broken.**
- `closed ‚Üí open` not logging a distinct `activity_log` action (`bug.reopened` vs `bug.status_changed`) ‚Üí regression metric impossible to compute.
- `resolved ‚Üí closed` without a verification gate ‚Üí any user can close a resolved Bug, bypassing QA.
- Simultaneous status updates from two users on the same Bug ‚Üí last-write-wins without optimistic locking means the earlier transition is lost from the audit trail.

**Terminal / forbidden states to guard.**
- No transition should skip intermediate states unless explicitly designed ‚Äî `open ‚Üí closed` directly (without `in_progress` + `resolved`) should be admin-only.
- `open ‚Üí resolved` without `in_progress` ‚Äî should this be allowed? Feature-map doesn't enumerate, but it's an implicit shortcut.

**How corruption would be detected.** Bug count in heatmap disagrees with raw `COUNT(*)` on `bugs` for a module. A Bug with `jira_sync_status=synced` but a stale `external_id` pointing to a deleted Jira issue.

---

### 3.3 `workspace_invites.status` ‚Äî Invite Lifecycle

**Why the transitions matter.** Invites expire after 24h via signed-token enforcement (not row-lifecycle). A revoked invite that still works via the signed token is a security hole ‚Äî someone retains access after admin removal.

**Transitions most likely to be broken.**
- `revoked` ‚Üí token still valid for the remainder of the 24h window ‚Üí the signed token check doesn't query the row's status, it only verifies the signature.
- `expired` email sent, but the token's 24h window hasn't actually elapsed due to clock skew between the signing server and the verification server.

**How corruption would be detected.** A user clicking a revoked invite link successfully joins the workspace. The `workspace_members` row is created from a `revoked` invite's `workspace_id`.

## 4. Silent Killers ‚Äî Automated Processes

Bunkai's MVP has five automated processes with no UI feedback path. A failure in any of them rots silently until someone notices the symptom ‚Äî usually days later.

### 4.1 Run-timeout sweeper (cron, every 15 min)

**What it does.** Marks Runs `running` for > `max_duration` (default 4h) as `aborted` with reason "executor timeout". Prevents agent-abandoned Runs from polluting active-run counts forever.

**What breaks if it fails.** Dashboard "Active Runs" counter drifts upward indefinitely. Agent-claimed Runs block re-execution (agent retries with same Idempotency-Key and gets the stale `run_id`). Heatmap counts are unaffected (terminal Runs don't feed the heatmap anyway), but the UX signal is wrong.

**How failure is detected today.** None ‚Äî the sweeper is Vercel-cron-triggered. If the cron silently fails (Vercel misconfiguration, function timeout), no alert fires.

**QA strategy.** After completing the MVP, seed 5 Runs in `running` status with `started_at > 4h ago` ‚Üí wait 16 minutes ‚Üí query all 5 ‚Üí status must be `aborted`, `abort_reason = "executor timeout"`. Validate the cron fires on Vercel's dashboard logs.

### 4.2 Jira bug-sync retry (cron, every 5 min)

**What it does.** Retries bugs with `jira_sync_status=failed` using exponential backoff. The Jira issue may have been manually deleted on the Jira side between retries.

**What breaks if it fails.** Bugs accumulate with `jira_sync_status=failed` and never get pushed to Jira. The QA lead (Mateo) expects Jira parity and opens a duplicate manually ‚Äî no automated fallback exists.

**How failure is detected today.** Only by viewing a bug's `jira_sync_status` field. No dashboard widget surfaces "unsynced bug count."

**QA strategy.** Simulate Jira 503 during bug creation ‚Üí verify `jira_sync_status=failed`. Mock Jira to return 200 on the next retry window ‚Üí verify the cron updates the Bug to `jira_sync_status=synced` and writes `external_id`.

### 4.3 Idempotency-key cleanup (cron, hourly)

**What it does.** Deletes rows from `idempotency_keys` where `expires_at < now()`. Bounded table size; {{PROJECT_KEY}}-037 guarantees 24h replay window, not infinite.

**What breaks if it fails.** The `idempotency_keys` table grows unbounded. At scale (hundreds of agents posting runs nightly), this bloats every `POST` latency (middleware checks the table on every write). Eventually the table scan dominates request time.

**How failure is detected today.** None ‚Äî no alert on table row count.

**QA strategy.** After a load test, verify `idempotency_keys` row count decreases within 25 hours of the oldest key's creation. If using Vercel cron, validate the execution log shows successful runs.

### 4.4 Materialized view refresh ‚Äî `module_defect_stats` (trigger + nightly fallback)

**What it does.** Recomputes `defect_count` per module on every Bug INSERT. Nightly full refresh as a safety net.

**What breaks if the trigger fails.** The defect heatmap renders stale counts. A bug filed 10 minutes ago doesn't appear in the module's defect tally ‚Äî Mateo's dashboard lies. The nightly refresh catches it, but within the window, decisions based on the heatmap are wrong.

**How failure is detected today.** Visual only ‚Äî the heatmap count disagrees with a manual `GET /bugs?module_id=` count. No automated reconciliation query.

**QA strategy.** File a Bug on module `/checkout/cart` ‚Üí within 5s, `GET /projects/{id}/defect-heatmap` includes the module with the updated count. Kill the trigger (simulate), file another Bug ‚Üí nightly refresh should still correct the count by morning.

### 4.5 `tsvector` index update (trigger on ATC INSERT/UPDATE)

**What it does.** Maintains the GIN index on ATC `title` + `tags` for full-text search and autocomplete ({{PROJECT_KEY}}-011 + {{PROJECT_KEY}}-031).

**What breaks if it fails.** ATC search returns partial or empty results. Users can't find ATCs by keyword ‚Äî they resort to manual tree navigation. The command palette (`Cmd+K`) returns no ATC hits.

**How failure is detected today.** Only when a user searches for a known ATC title and gets zero results.

**QA strategy.** Create ATC "Validate payment with Stripe webhook signature" ‚Üí search `GET /atcs?q=stripe` ‚Üí ATC appears. Edit the title to "Validate Stripe webhook signature validation" ‚Üí search `GET /atcs?q=stripe` ‚Üí ATC appears under new title. Search `GET /atcs?q=webhook+signature` ‚Üí ATC appears (multi-token query).

## 5. External Integrations ‚Äî Failure Points

### 5.1 Supabase Auth (OAuth + Magic Link)

**Business flow if down.** No user can sign in. Existing sessions with valid JWT continue for up to 1h. New sessions are impossible ‚Äî the adoption funnel is blocked.

**Timeouts and retry boundaries.** OAuth: browser-level redirect, ~10s user-perceived timeout before fallback suggestion. Magic link: email delivery latency, ~30s before UI shows "check your spam" (Journey 1 alt-path).

**Acceptable degradation.** Users with valid sessions can continue working. Magic-link fallback is offered within 30s of OAuth failure. If **both** OAuth providers (GitHub + Google) and magic-link are down simultaneously, the system is hard-fail for new sign-ins.

**Known quirks.** GitHub OAuth is frequently blocked by corporate IT proxies ‚Üí magic-link is the primary enterprise fallback. Supabase Auth email rate limits may delay magic-link delivery under load.

### 5.2 Jira REST API (Import + Bug Sync)

**Business flow if down.** Jira import jobs fail with `status: failed` and `errors[]` in the poll response ‚Äî user sees the failure, retries. Bug sync fails with `jira_sync_status=failed` ‚Äî the bug remains fully usable in Bunkai natively, Jira is **best-effort, not blocking**.

**Acceptable degradation.** Import and bug sync are async with user-visible status ‚Äî they degrade gracefully. The only scenario where Jira unavailability blocks a core flow is if the user **only** authors US/AC via import (no manual creation) and Jira is down ‚Üí they can't populate their workspace.

**Known quirks.** Jira REST `/search` pagination is offset-based and caps at 100 results/page. Large projects (5,000+ issues) require 50+ paginated calls ‚Äî the import worker must handle 429 rate limits from Atlassian. Bug-sync worker creates a Jira issue and writes back the `external_id` ‚Äî if the write-back fails, the `jira_sync_status` stays `syncing` forever until the retry cron corrects it.

### 5.3 Cloudflare R2 (Evidence Storage)

**Business flow if down.** Signed-URL generation 5xx ‚Üí step still records pass/fail, but `evidence_url` is empty. User sees a notification "evidence upload unavailable." R2 unreachable at view time ‚Üí broken-image placeholder in the run history.

**Acceptable degradation.** Run execution continues without evidence. The step result is durably stored ‚Äî evidence is a best-effort attachment, not a blocking dependency.

**QA strategy.** Simulate R2 503 on signed-URL generation ‚Üí verify the step result POST still returns 200 with `evidence_url: null`, not a 500 cascade.

### 5.4 Vercel Platform

**Business flow if down.** The entire API is unavailable. No workaround ‚Äî Vercel is the sole hosting platform in MVP.

**Acceptable degradation.** None. This is a hard-fail for everything.

### 5.5 Supabase Realtime

**Business flow if down.** Real-time updates stop. The runner UI falls back to polling (if implemented). Dashboards show stale data until manual refresh. The system is still functional but the "live" part of the UX is gone.

**Acceptable degradation.** All data mutations still work via REST ‚Äî nothing is lost, just delayed in visibility.

**Chain 1 ‚Äî Auth ‚Üí Everything.** If JWT verification in middleware fails, every gated endpoint returns 401. No user interaction is possible. The cascading blast radius is 100% of the system. This is why auth testing can never be deferred.

**Chain 2 ‚Äî ATC ‚Üí Test ‚Üí Run.** An ATC with broken `atc_acceptance_criteria` M:N join (the link is missing) creates a silent defect: the ATC appears in the library, attaches to a Test, starts a Run, but when the agent fetches `?expand=atcs.steps,atcs.assertions`, the response is incomplete. The agent executes the wrong assertions. The resulting run data is subtly wrong ‚Äî passes when it should fail, false negatives in the heatmap. This is the most dangerous cascade because it silences real defects.

**Chain 3 ‚Äî Bug ‚Üí Jira sync ‚Üí external_id write-back.** If `POST /bugs` succeeds but the Jira sync worker creates the Jira issue and then **fails to write back** `external_id` to the Bug row, the bug exists in both systems but is unlinked. Mateo assumes it wasn't synced, creates a duplicate manually. The data-model traceability (Bug ‚Üí Jira issue) is broken without any 500 error ‚Äî the Bug row just sits with `jira_sync_status=syncing`.

## 7. Edge Cases Developers Commonly Forget

**Concurrency.**
- Two users editing the same ATC simultaneously ‚Üí last write wins without version-check (no optimistic locking in spec). QA must verify that both users see the other's change after save, not silently clobber.
- Starting a Run on the same Test twice with the same `Idempotency-Key` within the same second ‚Üí duplicate detection must gate at the DB constraint level, not application level.

**Data limits.**
- Module depth exactly 6 ‚Üí succeeds. Depth 7 ‚Üí 400. What about moving a module at depth 6 to depth 3 (changing parent)? The new depth should be recalculated on the parent path, not the old path.
- Jira import: 500 issues (exactly the cap) ‚Üí all succeed. 501 issues ‚Üí chunked, first chunk succeeds, second chunk rejected with error, or full 400?
- Markdown body: User Story `description` at exactly 50KB ‚Üí accepted. 50KB + 1 byte ‚Üí 400 with `BODY_TOO_LARGE`.

**Timezone / DST.**
- `started_at` / `finished_at` stored as ISO 8601 with timezone. A Run started at 01:30 during DST fall-back (clocks go back) ‚Üí duration calculation must not go negative.
- Sweeper cron runs at 15-min intervals ‚Äî during DST spring-forward, one cron invocation is skipped (02:00 ‚Üí 03:00). A Run started at 01:50 could exceed `max_duration` before the sweeper runs at 03:15.

**Permission boundaries.**
- `member` creates an ATC, gets demoted to `viewer` ‚Üí can they still edit that ATC? RLS should block write but the UI might cache the old permission.
- PAT issued by a user who is later removed from the workspace ‚Üí token should still authenticate (`user_id` still valid) but every RLS-gated query should return empty/403 because `workspace_members` row is missing.
- Multi-workspace user: active workspace is workspace A but the API call targets workspace B's entity ‚Üí RLS on workspace B's table should reject even though the user HAS access to workspace B (it's just not the **active** workspace ‚Äî middleware context should enforce active workspace, not just membership).

**Orphaned states.**
- Module soft-deleted (archived) ‚Üí all descendant US, AC, ATC, Tests cascade-archived. But Runs on those Tests **still exist** and are still queryable via `GET /runs?test_id=`. What happens when the Run history references an archived Test?
- ATC duplicated, then the source ATC is archived ‚Üí the copy survives. No FK violation because duplication creates a new row.

**Idempotency.**
- Idempotency-Key replay with a **different payload** but same key ‚Üí should return the cached response (original payload wins) OR 409 `IDEMPOTENCY_KEY_MISMATCH`? Spec says "cached response" ‚Äî QA must verify this behavior.
- Idempotency key exactly 64 characters ‚Üí accepted. 65 characters ‚Üí 400 `IDEMPOTENCY_KEY_TOO_LONG`.

## 8. Pre-Release Checklist

1. Verify every authenticated endpoint returns 401 with explicit `code` (`TOKEN_EXPIRED`, `TOKEN_REVOKED`, `INVALID_TOKEN`) for each auth failure mode ‚Äî not a generic 401 without a code.
2. Verify `POST /atcs` rejects an ATC whose `acceptance_criterion_ids` includes an AC belonging to a different User Story ‚Äî 400 with `AC_MISMATCH_US`, not a silent mis-link.
3. Verify `POST /runs` with `Idempotency-Key` returns the cached `run_id` on exact replay within 24h, even after the original Run has been aborted.
4. Verify `POST /runs/{id}/steps/{stepId}/result` on a terminal Run returns 409 `RUN_TERMINAL` ‚Äî not a 200, not a 500.
5. Verify RLS cross-tenant isolation: user in workspace A cannot `GET` workspace B's entities by ID (403, never 200 with data).
6. Verify token revocation is immediate (within the same second): `DELETE /auth/tokens/{id}` ‚Üí next request with that token ‚Üí 401 `TOKEN_REVOKED`.
7. Verify ATC edit propagation: edit an ATC linked to 3 Tests ‚Üí all 3 Tests reflect the change when fetched via `?expand=atcs.steps`.
8. Verify Realtime: file a Bug from the Run runner ‚Üí defect heatmap in a separate browser tab updates within 5s without refresh.
9. Verify module tree depth guard: depth 6 OK, depth 7 ‚Üí 400. Circular parent reassignment ‚Üí 409 `MODULE_CIRCULAR_PARENT`.
10. Verify Jira import dedup: import the same JQL twice ‚Üí second run imports 0 new issues, `imported_count=0`, `status=succeeded`.
11. Verify run-timeout sweeper: seed a `running` Run with `started_at > 4h ago` ‚Üí within 15 minutes, Run transitions to `aborted` with reason "executor timeout".
12. Verify ATC version counter increments on edit and is reflected in `GET /atcs/{id}` response.
13. Verify `PATCH /modules` move operation recalculates materialized `path` for the moved module AND all descendants.
14. Verify soft-delete cascade: archive a Module ‚Üí descendant US/AC/ATC/Tests all get `archived_at` set. Listing endpoints exclude them by default.
15. Verify complete Setup journey end-to-end with every auth variant (GitHub OAuth, Google OAuth, magic-link) ‚Äî no 500s, no blank screens, <10 minutes.

## 10. Discovery Gaps

These are things the business maps cannot answer today. They represent risk that should be resolved before GA ‚Äî each one has a resolution path.

1. **`imports` table does not exist in the canonical ERD** (data-map ¬ß6-G1). The import flow is documented in {{PROJECT_KEY}}-009 and architecture ¬ß4.4, and the API map (api-map ¬ß4.12) declares `POST /imports/jira` and `GET /imports/{import_job_id}`, but the backing table is absent from the ERD. I cannot verify how import state is persisted or what happens on worker crash mid-import. **Resolution**: materialize during `/project-bootstrap`.

2. **No U/D/L endpoints for User Stories and Acceptance Criteria in api-contracts.yaml v1.0** (feature-map ¬ß10-Gap #4, api-map ¬ß10-Gap #3). Only `POST` is documented. {{PROJECT_KEY}}-035 implies full CRUD but the spec is incomplete. An agent fetching the OpenAPI spec at runtime will assume these resources are create-only. **Resolution**: surface GET, PATCH, DELETE, and List endpoints before `/project-bootstrap` scaffold.

3. **No public API for Environments or Integrations management** (feature-map ¬ß10-Gap #3, api-map ¬ß10-Gap #2). Both tables exist in the ERD and surface in the Settings UI, but Bearer-accessible CRUD is absent. An AI agent or CLI user configuring Jira integration or adding an environment must use the browser UI ‚Äî headless operation depends on this. **Resolution**: confirm whether these should be Bearer-accessible or are settings-only via Server Actions.

4. **Run-timeout sweeper behavior is journey-implied, not functionally specified** (feature-map ¬ß10-Gap #5, api-map ¬ß10-Gap #5). User Journey 3 mentions a sweeper cron at 4h default, but no FR documents it. I cannot verify the exact abort condition, the `max_duration` configurability, or whether the sweeper handles the race condition (Run finishes between sweeper's SELECT and UPDATE). **Resolution**: codify as explicit FR before GA.

5. **Access-token `last_used_at` semantics are undefined** (data-map ¬ß9-G6). {{PROJECT_KEY}}-034 describes issuance, revocation, and scopes, but `last_used_at` tracking is not specified. Token-rotation UX (showing "last used 3 days ago" in the token list) depends on this. **Resolution**: confirm `last_used_at` update on every auth lookup during `/project-bootstrap`.

6. **Run `executor.identity` is free-text, not an FK** (data-map ¬ß9-G7). For human runs this should reference `auth.users.id`; for agent runs it's a label like `claude-code-nightly-v3`. The polymorphic design means the executor column can't be FK-validated ‚Äî a typo in the agent label creates a Run with no attribution path. **Resolution**: decide during bootstrap whether to split into two columns or keep as string + type discriminator with validation.

7. **Bug state machine sub-transitions not enumerated in SRS** (data-map ¬ß9-G4). The ERD declares `bugs.status` exists, but `functional-specs.md` does not enumerate every allowed transition. Can a Bug go directly from `open` to `resolved` without `in_progress`? The heatmap tag `won't fix` is mentioned in the data-map state machine but not in any FR. **Resolution**: have `/product-management` enumerate all transitions before GA.

8. **Realtime channel auth for Bearer PAT not documented** (api-map ¬ß10-Gap #8). {{PROJECT_KEY}}-040 declares Realtime channels keyed by `project_id`, but the auth model covers REST only. How an agent using a Bearer PAT subscribes to Realtime WebSocket channels (vs browser JWT session) is not specified. **Resolution**: confirm MVP Realtime is browser-only (JWT), and agent push is REST-polling until the Phase-2 WebSocket protocol. If agents need Realtime, the auth model must be extended.

9. **Workspace invite lifecycle endpoints missing** (feature-map ¬ß10-Gap #2, api-map ¬ß10-Gap #1). No GET-list, resend, or revoke for invites. The 24h-expiry behavior means users will hit expired-invite cases, and there's no API path for an admin to resend or cancel. **Resolution**: surface invite management endpoints before GA ‚Äî essential for invite UX.

10. **Feature-map was generated without a live codebase** ‚Äî every feature is `Planned`, no WIP/Beta/Stable signals. The CRUD matrix and QA-relevance tags are derived from PRD/SRS, not from implementation reality. Once code lands, re-running `/business-feature-map` and this command will add breakage-likelihood signals from git history and actual test coverage.

11. **No live staging target yet** ‚Äî the target app (https://upexbunkai.vercel.app) is referenced in `.agents/project.yaml` but is greenfield. All flows in this plan are contract-first ‚Äî validated against the OpenAPI spec and SRS, not against a running instance. `/project-bootstrap` must create the first deployable cut before any of these tests can execute.
